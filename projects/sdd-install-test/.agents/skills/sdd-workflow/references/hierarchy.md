# 진실의 계층 — SDD 원칙

## 계층 구조

```
스티어링 (방향/원칙)       ← 가장 높은 권위
    ↓
  스펙 (요구사항 → 설계 → 태스크)
    ↓
  코드 (구현 결과물)        ← 결과물, 진실 아님
```

**코드는 항상 결과물이지 진실의 출처가 아니다.**
코드와 스펙이 다르면 → 코드를 수정한다. 스펙을 코드에 맞추지 않는다.

## 드리프트 발생 시 대응

| 드리프트 | 올바른 대응 | 잘못된 대응 |
|----------|-------------|-------------|
| 코드 ≠ 스펙 | 코드를 스펙에 맞게 수정 | 스펙을 코드에 맞게 변경 |
| 스펙 ≠ 스티어링 | 스펙 검토 또는 스티어링 의식적 갱신 | 자동 덮어쓰기 |
| 스티어링 변경 필요 | 명시적 결정 후 `--intent` 플래그 사용 | 코드 보고 자동 업데이트 |

## 스티어링 모드

### Bootstrap 모드
- 언제: `.sdd/steering/`이 비어있거나 새 프로젝트
- 동작: 코드베이스 분석 후 스티어링 파일 생성

### Sync 모드
- 언제: 스티어링 파일 있음, `--intent` 없음
- 동작: 드리프트 감지 및 보고. 코드를 스펙에 맞게 정렬 확인
- **스티어링을 코드에 맞춰 자동 업데이트하지 않음**

### Rewrite 모드 (`--intent "<새 방향>"`)
- 언제: 의식적으로 방향을 바꿀 때
- 동작: 새 방향 중심으로 스티어링 재작성
- 코드베이스는 도메인 지식만 참고, 기술 결정은 무시

## 스펙 계층 내부 흐름

```
requirements.md   ← WHAT: 테스트 가능한 요구사항 (EARS 형식)
    ↓
design.md         ← HOW: 아키텍처, 컴포넌트, 인터페이스 계약
    ↓
tasks.md          ← WHO/WHEN: TDD 태스크, 병렬 가능성, 시간 추정
    ↓
구현 코드          ← 결과물
```

각 단계는 상위 단계에서 파생된다. 하위 단계가 상위를 덮어쓰지 않는다.

## 스티어링 파일 역할

| 파일 | 권한 | 내용 |
|------|------|------|
| `steering/product/SKILL.md` | 제품 방향 | 목적, 핵심 기능, 타겟, 제약 |
| `steering/tech/SKILL.md` | 기술 결정 | 스택, 주요 결정, 외부 서비스 |
| `steering/structure/SKILL.md` | 구조 결정 | 폴더 패턴, 네이밍, 모듈 경계 |

## 갱신 트리거

**스티어링을 업데이트해야 할 때:**
- 비즈니스 방향이 바뀌었을 때
- 기술 스택을 전환하기로 결정했을 때
- 갈아엎기 전 새 방향 정의할 때 (`--intent`)

**스티어링을 업데이트하면 안 될 때:**
- 매 코드 변경마다
- 버그 수정 시
- 아키텍처 변경 없는 리팩토링 시
