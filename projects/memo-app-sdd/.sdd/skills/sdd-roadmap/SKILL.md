# 프로젝트 로드맵

## 언제 만드는가

**필요할 때 (선택):**
- 스펙이 3개 이상 예상되는 프로젝트
- 팀 협업 또는 장기 프로젝트
- 스펙 간 의존성이 있을 때

**불필요한 경우:**
- PoC, 실험, 단일 스펙 프로젝트
- 결과물이 불명확한 탐색 단계

## 핵심 원칙

**지금 아는 것만 적는다.** 처음부터 완전한 로드맵을 만들려 하지 말 것.  
5줄짜리 초안도 괜찮다. 스펙이 쌓이면서 자연스럽게 성장한다.

**로드맵은 계약이 아니다.** 방향을 공유하는 문서다.  
완료된 것은 체크하고, 새로 발견된 것은 추가한다.

## roadmap.md 구조

```markdown
# 로드맵 — [프로젝트명]

> 마지막 업데이트: YYYY-MM-DD

## 목표
[이 프로젝트가 완성됐을 때 무엇이 가능한가 — 1~3문장]

## 마일스톤

### M1: [이름] [상태: 진행중|완료|예정]
> [이 마일스톤의 목적 — 1문장]
- [ ] 스펙: `TIMESTAMP-feature-a` — [한 줄 설명]
- [x] 스펙: `TIMESTAMP-feature-b` — [한 줄 설명]

### M2: [이름] [예정]
> [목적]
- [ ] 스펙: [아직 미정] — [아이디어 수준도 OK]

## 의존성
- M2는 M1 완료 후 시작
- feature-c는 feature-a API 필요

## 현재 위치
M1 진행 중 — feature-b 완료, feature-a 작업 중
```

## Init 모드 (신규 로드맵)

1. `.sdd/specs/` 디렉토리 읽기 (기존 스펙 확인)
2. 스티어링 파일 읽기 (product 목적 파악)
3. 알려진 것만으로 최소한의 로드맵 작성
4. 불확실한 부분은 `[아직 미정]`으로 표시
5. `.sdd/roadmap.md`에 저장

## Update 모드 (기존 로드맵 갱신)

1. `.sdd/roadmap.md` 읽기
2. `.sdd/specs/` 디렉토리로 완료된 스펙 확인
3. 완료 항목 `[x]` 체크
4. 새로 발견된 스펙/기능 추가
5. "현재 위치" 업데이트
6. 저장
