# TDD 프로토콜 상세 — 오류 처리 및 자율 모드 규칙

## 오류 발생 시 복구 절차

### 1단계: 오류 분류

**가벼운 오류 (재시도 가능)**:
- 테스트 실패 (구현 버그)
- 타입 오류
- 임포트 경로 오류

**심각한 오류 (태스크 스킵)**:
- 3회 재시도 후도 실패
- 외부 서비스 불가 (네트워크 등)
- 다른 태스크에 영향 없을 때

**치명적 오류 (중단)**:
- 다른 모든 태스크를 블로킹하는 실패
- 데이터 손실 위험
- 보안 위험

### 2단계: 복구 시도 (최대 3회)

```
1회: 오류 메시지 분석 → 명백한 수정
2회: 스펙/설계 재확인 → 접근법 변경
3회: 스티어링 재확인 → 제약 내 최선의 방법
실패 시: 로그에 기록, 다음 태스크로 계속
```

### 3단계: 로그 기록

```
[HH:MM] ⚠️ 태스크 N.M 오류: [오류 메시지 요약] — 재시도 1/3
[HH:MM] ⚠️ 태스크 N.M 오류: [수정 시도] — 재시도 2/3
[HH:MM] ❌ 태스크 N.M 실패 (3회 시도): [최종 오류] — 스킵
```

## 모호한 요구사항 처리

1. requirements.md에서 가장 합리적인 해석 선택
2. 해석 내용 로그에 기록:
   ```
   [HH:MM] 🤔 결정: [모호한 부분] → [선택한 해석] (이유: [간단한 이유])
   ```
3. 구현 계속 진행

## 자율 모드 (overnight) 추가 규칙

### 계속 진행하는 경우
- 단일 태스크 실패 (다른 태스크 독립적)
- 경고 수준 오류
- 모호한 요구사항 (최선의 해석으로 진행)

### 중단 후 로그에 기록하는 경우
- 모든 후속 태스크가 실패한 태스크에 의존
- 치명적 오류 발생
- 루트 설정 파일 없음 (spec.json, requirements.md 등)

## 스티어링 가드레일 체크

구현 시작 전 확인:
- [ ] 사용하려는 라이브러리가 `tech` 스티어링에 있거나 허용됨
- [ ] 파일 생성 위치가 `structure` 스티어링을 따름
- [ ] 네이밍 규칙이 `structure` 스티어링을 따름
- [ ] 스티어링과 충돌하는 선택을 하려 한다면 → 스티어링 우선

## 테스트 격리 원칙

- 각 테스트는 독립적으로 실행 가능해야 함
- 테스트 간 공유 상태 없음
- 외부 서비스는 모킹
- 데이터베이스는 테스트 DB 또는 in-memory 사용

## 완료 요약 형식 (자율 실행 후)

```markdown
## 완료 요약
- 실행 시간: HH:MM ~ HH:MM (총 Xh Ym)
- 성공: N/M 태스크
- 실패: N개
- 결정사항: N개
- 다음 필요 작업:
  - [실패한 태스크 목록]
  - [수동 확인 필요 항목]
```
