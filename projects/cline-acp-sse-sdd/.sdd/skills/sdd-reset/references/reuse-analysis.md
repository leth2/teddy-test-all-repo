# 재사용 분석 상세 가이드

## 판단 기준

### 완료율 기반 판단

```
tasks.md 완료율:
- 80% 이상 완료 → 대부분 살릴 것
- 50~80% 완료 → 선택적 재사용
- 50% 미만 완료 → 대부분 버릴 것
```

### 요구사항 재사용 판단

살릴 요구사항:
- 비즈니스 목적이 변하지 않은 것
- AC(수락 기준)가 여전히 유효한 것
- 새 방향과 충돌하지 않는 것

버릴 요구사항:
- 리셋 이유가 된 요구사항
- 새 기술 스택에서 전제가 달라진 것
- 이미 폐기된 비즈니스 케이스

### 설계 재사용 판단

살릴 설계:
- 독립 컴포넌트 (다른 것에 의존하지 않음)
- 새 아키텍처에서도 동일한 역할의 컴포넌트
- 외부 서비스 통합 정보 (API 스펙, 인증 방식)

버릴 설계:
- 리셋 원인이 된 아키텍처 결정
- 버릴 기술 스택에 묶인 컴포넌트
- 새 방향에서 재정의될 인터페이스

## 분석 결과 출력 형식

```markdown
## 재사용 분석 결과

### 살릴 것 (새 스펙에 참고 권장)
**완료된 태스크:**
- [x] 1.1 사용자 모델 구현 — 재사용 가능
- [x] 1.2 DB 스키마 생성 — 스키마는 유지, 쿼리는 재작성

**유효한 요구사항:**
- 1.1 사용자 인증 요구사항 — 방향 동일
- 2.1 프로필 조회 요구사항 — 여전히 유효

**재사용 가능한 설계:**
- 사용자 데이터 모델 (User 엔티티)
- 외부 OAuth 연동 스펙

### 버릴 것 (새 스펙에서 새로 작성)
**미완료 태스크:**
- [ ] 3.1 대시보드 구현 — 방향 변경으로 재작성
- [ ] 3.2 알림 시스템 — 새 아키텍처로 재설계 필요

**무효한 요구사항:**
- 3.x REST API 요구사항 — GraphQL로 전환 결정

**폐기할 설계:**
- REST API 엔드포인트 설계 — GraphQL 스키마로 대체
```

## 아카이브 복원 방법

아카이브된 스펙 복원이 필요할 때:

```bash
# 아카이브 목록 확인
ls .sdd/archive/

# 특정 아카이브 복원
cp -r .sdd/archive/2026-03-13-09-30/001-auth/ .sdd/specs/001-auth/
```

## 스티어링 리셋 (`--steering`)

스티어링도 리셋할 때:

살릴 스티어링 내용:
- 도메인 지식 (비즈니스 용어, 규칙)
- 검증된 외부 서비스 연동 정보
- 여전히 유효한 제약 조건

버릴 스티어링 내용:
- 버릴 기술 스택 결정
- 변경되는 폴더 구조 패턴
- 새 방향과 상충하는 제약

리셋 후: `/sdd:steering --intent "<새 방향>"` 으로 새 스티어링 작성
