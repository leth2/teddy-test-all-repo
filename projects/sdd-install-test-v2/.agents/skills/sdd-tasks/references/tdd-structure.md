# 태스크 TDD 구조 상세 및 예시

## 전체 tasks.md 구조 예시

```markdown
# 태스크: 1741834500-user-auth

## 진행 상황
- 전체: 8개 태스크
- 완료: 0개
- 예상 총 시간: ~14h

---

## 1. 인증 기반 설정
> 요구사항: 1.1, 1.2 | 예상: ~3h

- [ ] 1.1 테스트 환경 설정 — 테스트 러너 실행, 빈 스위트 통과
- [ ] 1.2 사용자 모델 테스트 작성 (실패 확인) — 테스트가 실패하는 것 확인
- [ ] 1.3 사용자 모델 구현 — 1.2 테스트 통과
- [ ] 1.4 사용자 모델 리팩토링 — 모든 테스트 유지, 코드 정리

## 2. 로그인 API (P)
> 요구사항: 2.1, 2.2 | 예상: ~4h | 병렬 실행 가능

- [ ] 2.1 로그인 엔드포인트 테스트 작성 (실패) — POST /auth/login 응답 형식 검증
- [ ] 2.2 로그인 로직 구현 — 이메일+비밀번호 검증, JWT 발급
- [ ] 2.3 오류 케이스 처리 — 잘못된 자격증명, 계정 잠금

## 3. 토큰 갱신 API (P)
> 요구사항: 2.3 | 예상: ~2h | 병렬 실행 가능

- [ ] 3.1 갱신 엔드포인트 테스트 작성 (실패)
- [ ] 3.2 갱신 로직 구현

## 4. 통합 테스트
> 요구사항: 1.1~2.3 | 예상: ~2h | requires: 2, 3 완료

- [ ] 4.1 E2E 인증 플로우 테스트
- [ ] 4.2 엣지 케이스 커버리지 확인
```

## TDD 사이클 세분화 원칙

### 언제 테스트를 별도 태스크로 분리하는가
- 복잡한 비즈니스 로직 구현 시 (항상)
- 외부 서비스 통합 시 (모킹 필요)
- 데이터 모델 생성 시

### 언제 통합 가능한가
- 간단한 유틸리티 함수 (테스트+구현 합쳐서 1시간 미만)
- 설정 파일 생성 (비즈니스 로직 없음)

## 의존성 명시 패턴

```markdown
## 5. 이메일 발송 (P)
> 요구사항: 3.1 | 예상: ~2h

## 6. 알림 통합
> 요구사항: 3.2 | requires: 2 (JWT 필요), 5 완료
```

## 커버리지 검증 체크

태스크 생성 후 반드시 확인:

```
requirements.md 요구사항 목록:
1.1 → 태스크 1.2, 1.3 ✅
1.2 → 태스크 1.4 ✅
2.1 → 태스크 2.1, 2.2 ✅
2.2 → 태스크 2.3 ✅
2.3 → 태스크 3.1, 3.2 ✅
3.1 → 태스크 5 ✅
⚠️ 3.2 → 매핑 없음! → 태스크 추가 필요
```

## 완료 기준 작성 지침

좋은 완료 기준:
- "테스트가 통과하는 것 확인"
- "모든 엣지 케이스 테스트 통과"
- "커버리지 80% 이상"
- "기존 테스트 회귀 없음"

나쁜 완료 기준:
- "구현함" (너무 모호)
- "코드 작성 완료" (검증 기준 없음)
