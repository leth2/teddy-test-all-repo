# 스펙 리셋/아카이브

## 단계

### 1. 대상 확인
- 대상 없으면 `.sdd/specs/` 전체 목록 표시 후 선택 요청
- `--steering` 있으면 `.agents/skills/project-steering/`도 대상에 포함

### 2. 재사용 분석 (`--skip-analysis` 없을 때)

**살릴 것 (재사용 권장)**:
- ✅ 완료된 태스크 (`[x]`) — 이미 구현된 기능
- ✅ 검증된 요구사항 — 방향이 바뀌어도 유효한 비즈니스 요구사항
- ✅ 아키텍처 결정 — 재사용 가능한 기술 스택/인프라 결정
- ✅ 완료된 설계 컴포넌트 — 재활용 가능한 모듈/인터페이스
- ✅ 통과한 테스트 케이스

**버릴 것 (새로 작성)**:
- ❌ 미완료 태스크 (`[ ]`)
- ❌ 방향이 바뀐 요구사항
- ❌ 잘못된 설계 결정
- ❌ 구현 중 발견된 기술 부채
- ❌ 오래된 스티어링 (`--steering` 포함 시)

### 3. 아카이브 실행

저장 경로:
```
.sdd/archive/
  YYYY-MM-DD-HH-MM/     ← 아카이브 시점 (분 단위)
    001-auth/            ← 원본 폴더 그대로 이동
    steering/            ← --steering 포함 시
```

- **삭제 없음**: 파일 이동 (복원 가능)
- 같은 스펙 여러 번 리셋 → 타임스탬프로 구분

### 4. 로그 기록

`.sdd/logs/YYYY-MM-DD.md`에:
```
[HH:MM] 🗄️ $FEATURE 아카이브됨 → .sdd/archive/YYYY-MM-DD-HH-MM/
[HH:MM] 📋 재사용 자산: [항목들]
[HH:MM] 🗑️ 폐기 대상: [항목들]
```

## 출력

1. 재사용 분석 결과 (살릴 것 / 버릴 것)
2. 아카이브된 파일 목록 및 위치
3. 다음 단계: `/sdd:spec-plan <새 설명>`
4. 살릴 것들을 새 스펙에 반영하는 방법 안내
