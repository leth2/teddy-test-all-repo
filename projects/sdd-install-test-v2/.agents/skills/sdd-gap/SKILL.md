# 프로젝트 갭분석

## spec-validate / spec-sync 와의 차이

| 도구 | 범위 | 목적 |
|------|------|------|
| spec-validate | 스펙 1개, 단계별 | 각 단계 품질 검증 |
| spec-sync | 스펙 1개 | 삼각형 3꼭짓점 동기화 |
| **spec-gap** | **프로젝트 전체** | **로드맵 vs 현실 빈틈** |

## 갭분석 레이어

### Layer 1: 로드맵 갭
`.sdd/roadmap.md`가 있을 경우:
- 로드맵에 있는데 스펙이 없는 기능 → ⚠️ 스펙 미작성
- 스펙은 있는데 로드맵에 없는 기능 → ⚠️ 로드맵 누락

### Layer 2: 스펙 커버리지 갭
`.sdd/specs/` 전체 스캔:
- requirements.md는 있는데 design.md 없음 → ⚠️ 설계 누락
- design.md는 있는데 tasks.md 없음 → ⚠️ 태스크 미분해
- tasks.md에 미완료(`[ ]` 또는 `[~]`) 태스크 있음 → 📋 미구현 목록

### Layer 3: 테스트 커버리지 갭
각 스펙의 requirements.md AC(수락 기준) 수 vs 테스트 파일 유무:
- AC가 있는데 테스트 파일이 없는 스펙 → ⚠️ 테스트 누락

### Layer 4: 스펙 간 의존성 갭
design.md에서 다른 스펙의 기능을 참조하는 경우:
- 참조된 스펙이 완료됐는가?
- 참조된 스펙이 존재하는가?

## 출력 형식

```
## 갭분석 결과 — [프로젝트명] [timestamp]

### 📊 전체 현황
- 총 스펙: N개
- 완료: N개 / 진행 중: N개 / 미시작: N개

### 🗺️ 로드맵 갭
- ⚠️ 스펙 미작성: [로드맵 항목들]
- ⚠️ 로드맵 누락: [스펙 목록]

### 📋 스펙 커버리지 갭
- ⚠️ 설계 누락: [스펙명]
- ⚠️ 태스크 미분해: [스펙명]
- 🔄 미완료 태스크: [스펙명: 태스크 수]

### 🧪 테스트 갭
- ⚠️ 테스트 파일 없음: [스펙명]

### 🔗 의존성 갭
- ⚠️ 미완료 의존: [스펙A → 스펙B (미완료)]

### ✅ 권장 다음 액션
1. [우선순위 순]
```

## 실행 시점

- 스프린트/마일스톤 시작 전
- `roadmap --update` 후
- "지금 뭘 해야 하지?" 감이 안 올 때
- 장기 프로젝트 주간 리뷰
