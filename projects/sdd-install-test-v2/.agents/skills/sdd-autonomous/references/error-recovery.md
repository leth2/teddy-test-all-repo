# 오류 복구 프로토콜 상세

## 오류 심각도 분류

### 가벼운 오류 (계속 진행)
- 테스트 실패 → 수정 후 재시도 (최대 3회)
- 파일 없음 → 기본값으로 생성 후 계속
- 타입 오류 → 수정 후 계속
- 임포트 경로 오류 → 수정 후 계속
- 경고 메시지 (오류 아님)

**처리**: 수정 시도, 최대 3회 재시도, 실패 시 스킵 후 계속

### 심각한 오류 (태스크 스킵)
- 3회 재시도 후도 실패
- 외부 서비스 불가 (네트워크 등)
- 다른 태스크에 영향 없는 경우

**처리**: 로그에 기록 (`❌`), 다음 독립 태스크로 계속

### 치명적 오류 (중단)
- 다른 모든 태스크를 블로킹하는 실패
- 데이터 손실 위험
- 보안 위험 (자격증명 노출 등)
- 루트 설정 파일 없음 (spec.json, requirements.md 없음)

**처리**: 로그에 기록 (`❌ FATAL`), 즉시 중단, 브리핑 생성 권장

## 재시도 전략

```
1회 재시도: 오류 메시지 분석 → 명백한 수정 (오타, 경로 등)
2회 재시도: 스펙/설계 재확인 → 접근법 변경
3회 재시도: 스티어링 재확인 → 제약 내 최선의 방법
이후: 스킵 결정
```

## 로그 형식 (상세)

| 아이콘 | 의미 |
|--------|------|
| ✅ | 성공적 완료 |
| ⚠️ | 경고 (계속 진행) |
| ❌ | 오류 (처리 필요) |
| 🤔 | 결정 (모호한 상황에서 판단) |
| 🗄️ | 아카이브 |
| 🚀 | 시작 |
| 🏁 | 완료 (전체 실행) |
| 🔴 | Red 단계 (테스트 작성) |
| 🟢 | Green 단계 (구현) |
| 🔵 | Refactor 단계 |

## 재시작 복구

overnight 실행이 중단된 후 재시작 시:

1. `tasks.md` 읽기 → `[x]` 완료된 태스크 파악
2. 로그 파일 읽기 → 중단 지점 파악
3. 마지막 완료 태스크 다음부터 시작
4. 완료 로그에 "재시작" 엔트리 추가:
   ```
   [HH:MM] 🚀 재시작: 태스크 N.M부터 (이전 실행: YYYY-MM-DD HH:MM 중단)
   ```

## 모호한 상황 결정 원칙

요구사항이 불명확할 때:
1. requirements.md에서 가장 합리적인 해석 선택
2. 스티어링과 일치하는 방향 우선
3. 더 단순한 구현 우선 (YAGNI)
4. 해석 내용 반드시 로그에 기록

설계가 불완전할 때:
1. design.md의 유사 컴포넌트 패턴 참고
2. `structure` 스티어링 따름
3. 결정 로그에 기록

## 블로킹 의존성 처리

태스크 A가 실패했고, 태스크 B가 A에 의존할 때:

```
[HH:MM] ❌ 태스크 N.M 실패 — 이 태스크에 의존하는 태스크: X, Y, Z
[HH:MM] ⚠️ 태스크 X, Y, Z 스킵 (태스크 N.M 블로킹)
[HH:MM] ✅ 독립 태스크 Q 계속 진행
```

## 외부 서비스 실패 처리

외부 API/서비스 연동 시 실패:
1. 모킹으로 대체 가능한지 확인
2. 모킹 가능 → 모킹으로 구현 후 로그에 기록
3. 모킹 불가 → 태스크 스킵, 로그에 "수동 설정 필요" 기록
