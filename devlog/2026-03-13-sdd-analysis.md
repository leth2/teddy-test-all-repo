# SDD 도구 실험: 문서화가 코드 품질을 보장하는가?

> 2026-03-13 | teddy-sdd 개발 기록

---

## 실험 배경

Spec-Driven Development(SDD) 도구를 만들었다. 이 도구는 AI 코딩 에이전트가 코드를 작성하기 전에 요구사항 → 설계 → 태스크를 문서화하도록 강제한다. 스티어링(방향) → 스펙(문서) → 코드 계층을 유지하면 코드 품질이 올라갈 것이라는 가설이었다.

실제로 그런지 실험해봤다.

---

## 실험 방법

같은 프로젝트를 세 가지 방법으로 만들었다.

**대상 프로젝트:** ACP(Agent Client Protocol) 브릿지 서버  
`claude-agent-acp` 프로세스를 생성하고 WebSocket으로 React UI와 연결하는 Node.js 서버.

**세 가지 버전:**
1. **직접 코딩** — SDD 없이 바로 구현
2. **SDD v1** — 기존 SDD 도구 적용 (요구사항 → 설계 → 태스크 → 구현)
3. **SDD v2** — 개선된 SDD 도구 적용 (v1 + 검증 단계 + 외부 스펙 참조)

---

## 결과

### 실제 통신 가능 여부

| 버전 | ACP 메서드명 | 핸드셰이크 | 실제 동작 |
|------|-------------|-----------|---------|
| 직접 코딩 | `prompt`, `newSession` (⚠️ 부분 틀림) | ✅ 있음 | ⚠️ 부분 동작 |
| SDD v1 | `message`, `newSession` (❌ 전부 틀림) | ❌ 없음 | ❌ 통신 불가 |
| SDD v2 | `session/prompt`, `session/new` (✅ 정확) | ✅ 있음 | ✅ 완전 동작 |

### 발견된 ACP 프로토콜 오류 (v1 대비 v2)

| v1 코드 | v2 수정 |
|---------|---------|
| `method: "message"` | `method: "session/prompt"` |
| `method: "newSession"` | `method: "session/new"` |
| 핸드셰이크 없음 | `initialize → session/new` 순서 강제 |
| notification에 id 부여 | id 없음 (JSON-RPC 표준 준수) |
| `sendPermissionResponse(approved: bool)` | `session/request_permission` response 형식 |

## 역설적인 발견

**SDD v1이 직접 코딩보다 나빴다.**

직접 코딩할 때 AI는 스스로 ACP 스펙을 찾아봤다.  
SDD v1에서 AI는 문서 생성에 집중하다가 프로토콜 조사를 빠뜨렸다.

결과적으로 "잘 문서화된 틀린 코드"가 만들어졌다.  
요구사항도 있고, 설계도 있고, 태스크도 있는데 — 실제로 에이전트와 통신이 안 됐다.

---

## 원인 분석

SDD 도구의 설계 구멍이었다.

- 요구사항 생성 단계: "무엇을 만들 것인가" ✅
- 설계 단계: "어떻게 만들 것인가" ✅
- **외부 프로토콜/API 스펙 조사: 없음** ❌

AI는 모르는 것을 모르는 채로 문서를 작성했다. ACP 프로토콜의 정확한 메서드명을 모르면서 설계 문서에 잘못된 인터페이스를 적었고, 그걸 그대로 구현했다.

직접 코딩에서는 이 단계가 암묵적으로 포함됐다.  
SDD에서는 명시적으로 강제하지 않으면 빠진다.

---

## 해결책 (v2에 적용)

```
설계 단계에서 외부 API/프로토콜 발견 시:
  1. WebSearch로 공식 스펙 검색
  2. 핵심 내용을 references/ 폴더에 저장
  3. 설계 문서에 참조 링크 추가
  4. 구현 전 references/ 먼저 읽기
```

v2에서 이 흐름으로 ACP 공식 스펙을 찾았고, 5개의 프로토콜 오류를 잡았다.

## 시사점

### 1. 문서화 ≠ 코드 품질 보장

SDD(또는 어떤 문서 기반 개발 방법론이든) 문서 생성 자체는 코드 품질을 보장하지 않는다. 문서가 틀리면 코드도 틀린다.

### 2. AI가 모르는 걸 모른다는 문제

AI는 자신이 모르는 외부 스펙을 그럴듯하게 추측한다. `newSession`이 맞는지 `session/new`가 맞는지 모르면 — 그냥 둘 중 하나를 선택한다. 확인하지 않는 한.

외부 API, 프로토콜, 라이브러리를 사용하는 모든 프로젝트에서 "실제 스펙 확인" 단계는 선택이 아니라 필수다.

### 3. 검증이 스펙 조사를 강제한다

v2의 설계 검증 단계가 "외부 스펙 참조 없음 → 웹서치 트리거"를 만들었다. 검증이 없었다면 v2도 v1과 같은 실수를 했을 것이다.

### 4. 정직한 상태 표시가 중요하다

v1의 tasks.md는 모든 태스크가 `[x]`(완료)였지만 실제 테스트는 실행되지 않았다.  
v2는 `[~]`(코드 작성됨, 테스트 미실행)으로 정직하게 표시했다.

거짓 완료 표시는 실제 프로젝트에서 심각한 문제로 이어진다.

---

## 결론

**SDD 도구가 도움이 되려면 세 가지가 필요하다:**

1. **외부 스펙 조사 강제** — 모르는 것을 모른 채 문서 쓰지 않기
2. **검증 단계** — 문서가 실제로 맞는지 확인
3. **정직한 완료 기준** — 테스트 통과 여부를 정확히 표시

문서화 자체는 수단이지 목적이 아니다.  
"잘 문서화된 틀린 코드"는 문서가 없는 것보다 나쁠 수 있다 — 틀렸다는 것을 알아차리기 더 어렵기 때문에.
