# SDD 삼각형 — 스펙, 테스트, 코드는 항상 함께 움직여야 한다

> 2026-03-13 | 원문: [The Spec Driven Development Triangle](https://www.dbreunig.com/2026/03/04/the-spec-driven-development-triangle.html)  
> Drew Breunig이 MLOps Community "Coding Agents" 컨퍼런스에서 발표한 내용

---

## 코드가 없는 오픈소스 라이브러리

Drew는 자전거를 타다가 이런 생각을 했어:

> "AI가 이렇게 코딩을 잘 한다면, 굳이 코드를 공유할 필요가 있을까?"

그래서 실험을 했어. **whenwords** — 코드가 없는 소프트웨어 라이브러리.

GitHub에 올린 것:
1. **스펙 문서** — "이 라이브러리가 뭘 해야 하는지" 설명 (Unix 타임스탬프를 "약 12시", "5시간 전" 같이 사람이 읽기 좋게 변환)
2. **750개 테스트** — "이 입력이 들어오면 이 결과가 나와야 해"
3. **install.md** — AI에게 붙여넣을 한 문단짜리 지시문

사용자가 하는 것: 원하는 언어(Python이든 Rust든)를 지정하고 AI에게 던져줌 → AI가 코드 생성.

결과: Karpathy가 좋아요를 눌렀고 GitHub에서 1,000개 이상의 별을 받았어.

더 흥미로운 건 — 실제 오픈소스처럼 돌아갔어. 누군가 PR을 올렸는데 이런 내용이었어:

> "이 테스트에서 기대하는 결과가 스펙의 반올림 규칙을 위반해요. 둘을 맞춰야 해요."

코드 기여가 아니라 **스펙과 테스트의 일관성을 고치는 기여**였어.

---

## 다른 팀들도 비슷한 실험을 했다

whenwords 이후 더 큰 프로젝트들이 같은 방식으로 시도됐어:

- **Vercel** — TypeScript로 Bash 환경 구현 (기존 Bash 테스트를 통과시키는 방식)
- **Pydantic** — Rust로 Python 인터프리터 구현 (Python 테스트를 통과시키는 방식)
- **Anthropic** — Claude 16개 + $20,000으로 C 컴파일러 구현 시도 (결국 실패에 가까웠지만)

---

## SDD 1세대에서 배운 것들

### 1. 테스트는 공짜가 아니다

모든 성공한 프로젝트는 이미 존재하는 방대한 테스트를 활용했어 (Bash 테스트, Python 테스트, C 테스트). 그게 없으면 테스트를 처음부터 만드는 게 너무 어렵고 비싸.

농담처럼 들리는 이 말이 점점 현실이 되고 있어:
> "앞으로 코드는 무료, 테스트는 유료가 될 것이다."

### 2. 빠르지만 완성은 아니다

AI로 코드 생성 속도는 폭발적으로 빠르지만, 프로젝트가 복잡해질수록 막히기 시작해. Anthropic의 C 컴파일러는 실패율 1%까지 갔다가 거기서 멈췄어. 버그 하나를 고치면 다른 게 깨지는 패턴이 반복됐어.

### 3. 구조가 점점 더 중요해진다

단순할 때는 AI가 아무 방향으로나 만들어도 돼. 하지만 복잡해질수록 "어떻게 쪼갤 것인가"가 결정적이야. 병렬로 작업할 수 있는 구조면 여러 AI가 동시에 달려들 수 있어.

---

## 가장 중요한 발견: SDD는 직선이 아니다

여기서 핵심이 나와.

많은 사람이 SDD를 이렇게 생각해:

```
스펙 작성 → AI에게 던짐 → 코드 완성
```

근데 실제로는 이래:

```
스펙 작성 → 코드 작성 → "스펙에 빈틈이 있었네" 발견
     ↑___________________________________|
```

**코드를 구현하면서 스펙이 개선된다.**

아무리 잘 만든 스펙이라도, 실제로 코드를 짜다 보면 "이 경우에 어떻게 해야 하지?"라는 질문이 생겨. 그게 스펙에 빠진 내용이야. 테스트가 잡지 못한 엣지 케이스야.

whenwords의 PR 스레드에도 이런 20개짜리 댓글 토론이 있었어 — 완벽한 스펙이었는데도.

## SDD 삼각형

Drew가 제안하는 개념이 **SDD 삼각형**이야:

```
        스펙
       /    \
      /      \
   테스트 ---- 코드
```

이 세 꼭짓점은 항상 **동기화** 상태여야 해.

- 스펙이 바뀌면 → 테스트도 바뀌어야 하고 → 코드도 바뀌어야 해
- 코드를 짜다가 새로운 케이스를 발견하면 → 스펙에 추가되어야 하고 → 테스트도 추가되어야 해
- 테스트가 통과 안 되면 → 코드 버그인지 테스트 버그인지 스펙 버그인지 판별해야 해

이 세 개가 따로 놀면 — "잘 문서화된 틀린 코드"가 만들어져.

---

## 역사적 맥락: 이미 겪은 문제다

Drew는 1963년 Margaret Hamilton 이야기를 꺼내. NASA Apollo 미션의 소프트웨어를 관리하던 그녀는 "소프트웨어 엔지니어링"이라는 단어를 처음 만든 사람이야.

당시 그녀가 직면한 문제:
- 코드가 너무 많아서 아무도 전체를 머릿속에 담을 수 없어
- 팀이 같이 작업하면 더 복잡해져
- 에러가 나면 우주선이 떨어져

지금 AI 코딩 시대의 문제:
- AI가 생성하는 코드가 너무 많아서 사람이 다 읽을 수 없어
- 여러 AI 에이전트가 동시에 작업하면 더 복잡해져
- 스펙과 코드가 따로 놀기 시작해

**우리는 60년 전 문제를 다른 형태로 다시 겪고 있어.**

그때 해결책이 버전 관리, 코드 리뷰, 테스트였다면 — 지금 해결책은 스펙/테스트/코드를 동기화하는 새로운 도구들이야.

## AI 시대의 GitHub은 어떤 모습이어야 할까?

Drew의 마지막 질문이 흥미로워:

지금 GitHub은 **코드**를 중심으로 만들어진 플랫폼이야. PR = 코드 변경.

하지만 SDD 세계에서 중요한 건 스펙과 테스트야. 코드는 그 결과물일 뿐이야.

앞으로의 협업 플랫폼은 이런 것들을 지원해야 할지도 몰라:
- 스펙 변경이 테스트와 코드에 어떤 영향을 미치는지 추적
- 세 꼭짓점의 동기화 상태를 한눈에 보기
- "코드 기여" 대신 "스펙 기여", "테스트 기여"도 동등하게 인정

---

## 정리하면

**SDD의 이상:** 스펙 + 테스트 → AI → 코드

**SDD의 현실:** 스펙 + 테스트 + 코드가 서로를 개선하는 피드백 루프

세 꼭짓점이 동기화를 유지해야 하고, 그게 무너지는 순간 프로젝트가 무너져.

오늘 teddy-sdd v2 실험에서 발견한 것 — "잘 문서화된 틀린 코드" — 이게 바로 SDD 삼각형이 어긋난 상태야. 스펙은 있었는데 코드가 외부 스펙을 반영하지 못했어. 삼각형의 한 꼭짓점이 빠진 거야.
