## 2. Claude Code 에이전트 팀 역공학

Claude Code v2.1.77 소스코드(12MB 번들)를 분석했다.

### 발견: 파일 기반 IPC

에이전트 팀의 통신 방식이 WebSocket이나 HTTP가 아니라 **순수 파일 I/O**라는 걸 확인했다.

```
~/.claude/teams/{team}/inboxes/{agent}.json
```

각 에이전트마다 JSON 파일 하나가 메시지 큐 역할을 한다. 메시지 보내기 = 상대 파일에 append, 읽기 = 자기 파일에서 `read: false` 필터. 동시성 제어는 `.lock` 파일.

**왜 파일 기반인가?**
- 프로세스 독립 — 에이전트가 죽어도 메시지 보존
- 디버깅 쉬움 — 파일 열면 바로 확인
- 구현 단순 — 소켓/HTTP 불필요

**메시지 전달 방식이 독특하다.** 소켓 폴링이 아니라 **LLM 턴 자체가 폴링 주기**다.

1. 에이전트가 응답을 생성하기 전 inbox 파일 읽기
2. 미읽 메시지를 XML로 변환
3. 시스템 프롬프트에 주입: `<teammate-message teammate_id="lead">...</teammate-message>`
4. LLM이 이걸 보고 다음 행동 결정
5. 도구 실행 (SendMessage, TaskCreate 등)

Claude의 대화 턴이 메시지 수신 타이밍인 셈. 폴링 없이도 동작한다.

**내부 코드네임:** tengu (天狗). 모든 텔레메트리 이벤트가 `tengu_` prefix다.

### 태스크 시스템

```
~/.claude/tasks/{team}/{id}.json
```

각 태스크도 개별 파일. Claim 메커니즘이 파일 락 기반이라 race condition이 없다.

```
lock 획득 → 파일 읽기 → owner 확인 → 소유 → lock 해제
```

### Demo Day Phase 2 — ACP connection.prompt() HANG 해결

밤새 붙잡고 있던 문제가 새벽에 해결됐다.

증상: `sendPrompt` 후 5분 타임아웃. `[ACP] prompt 완료` 로그 없음. 파일 미생성.

원인: `await this.connection.prompt()` — Cline v2.7.1이 prompt에 대한 completion 이벤트를 보내지 않아서 Promise가 영원히 pending.

해결: **fire-and-forget** 패턴으로 전환.

```typescript
// 변경 전 (HANG)
await this.connection.prompt({ sessionId, prompt: [...] });

// 변경 후 (fire-and-forget)
this.connection.prompt({ sessionId, prompt: [...] })
  .then(() => this.emit('session-done', sessionId))
  .catch((err) => this.emit('session-done', sessionId));  // 에러도 완료 처리
```

이후 Quinn 테스트에서 `[ACP] sendPrompt fire-and-forget 전송 완료` 로그 확인. HANG 없이 즉시 반환 성공. 파일 생성 여부는 계속 테스트 중.

### OpenClaw 게이트웨이 원격 접근 설정
