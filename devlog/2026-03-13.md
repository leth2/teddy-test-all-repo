## SDD 도구 개선 적용

### 신규
- `sdd-validate` 스킬 추가 — Requirements/Design/Tasks/Impl 4단계 검증 체크리스트
- `spec-validate` 커맨드 추가 — `/sdd:spec-validate <feature> [--phase ...]`

### 기존 스킬 강화
- `sdd-requirements` — 생성 후 자동 검증, 통과 후에만 approved 처리
- `sdd-design` — 외부 스펙 발견 시 WebSearch → `references/` 저장 흐름
- `sdd-tasks` — `[~]` 상태 추가 (코드 작성됨, 테스트 미실행), `[x]`는 테스트 통과 확인 필수
- `sdd-impl` — 구현 전 design.md 인터페이스 계약 확인, 완료 후 자동 검증

### 워크플로우 변화
```
전: 생성 → 승인 → 다음 단계
후: 생성 → 자동 검증 → 통과 시 승인 → 다음 단계
              ↓ 실패 시
         수정 → 재검증
```

## SDD v1 vs v2 비교 + 직접 코딩 비교

### 3개 버전 비교 결과

| 항목 | 직접코딩 POC | SDD v1 | SDD v2 |
|------|-------------|--------|--------|
| ACP 핸드셰이크 | ✅ 있음 | ❌ 없음 | ✅ 있음 |
| 메서드명 정확도 | ⚠️ 절반 (`prompt` 사용) | ❌ 틀림 (`message`) | ✅ 정확 (`session/prompt`) |
| JSON-RPC notification id | ✅ 없음 | ❌ 있음 | ✅ 없음 |
| 테스트 파일 | ❌ 없음 | ✅ 있음 (미실행) | ✅ 있음 (미실행, [~] 정직) |
| 외부 스펙 참조 | ❌ 없음 | ❌ 없음 | ✅ references/acp-protocol.md |
| 스펙 문서 | ❌ 없음 | ✅ req/design/tasks | ✅ req/design/tasks + validation |
| 실제 통신 가능 여부 | ⚠️ 부분 | ❌ 불가능 | ✅ 가능 |

### 핵심 발견

직접 코딩 POC가 SDD v1보다 코드 정확도가 높았음.
이유: 직접 코딩 시 AI가 실제로 ACP를 조사했지만, SDD v1에서는 문서 생성에 집중하다가 프로토콜 조사를 빠뜨림.

SDD v2가 세 버전 중 가장 정확함.
이유: 검증 단계가 외부 스펙 웹서치를 강제했고, references/에 저장된 스펙을 구현에서 참조함.

### 교훈

- SDD 없음 > SDD v1 (코드 정확도 기준)
- SDD v2 > SDD 없음 (코드 정확도 + 문서화 기준)
- **외부 스펙 references/가 SDD의 핵심 가치**
- 문서 생성만으로는 코드 품질 보장 안 됨 — 검증이 필수

### v2에서 발견된 ACP 프로토콜 오류 5개 (v1 대비)
1. `newSession` → `session/new`
2. `method: "message"` → `method: "session/prompt"`
3. `method: "permission"` → JSON-RPC Response 형식
4. `method: "permission-request"` → `session/request_permission`
5. id 없는 notification 처리 누락
