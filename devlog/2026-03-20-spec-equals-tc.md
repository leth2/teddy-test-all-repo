# Spec = TC — 스펙과 테스트가 하나가 되는 개발 패러다임

> 2026-03-20 | 리서치 노트

---

## 왜 테스트는 항상 나중에 쓰이나

TDD를 도입하겠다는 팀은 많다. 실제로 테스트를 먼저 쓰는 팀은 드물다. 이유는 단순하다. 테스트를 먼저 쓰려면 **구현 전에 시스템의 동작을 명확히 알아야 한다.** 그게 어렵기 때문에 코드부터 만들고 테스트를 나중에 붙인다.

AI 에이전트가 코드를 생성하는 환경에서는 이 문제가 역전된다. 에이전트가 코드를 만들어주니 "먼저 알고 있어야 하는 것"이 테스트와 스펙으로 더 명확하게 분리된다.

그리고 가장 흥미로운 통찰이 등장한다:

**스펙과 테스트 케이스는 같은 정보의 다른 표현이다.**

---

## Spec = TC 가 의미하는 것

"이 기능이 무엇을 해야 하는가"를 명세로 쓰면 스펙이다.
"이 기능이 제대로 동작하는가를 어떻게 확인하는가"를 코드로 쓰면 테스트 케이스다.

둘 다 **동일한 의도(Intent)** 를 표현한다. 형식만 다를 뿐이다.

현대적인 SDD 도구들이 지향하는 목표는 이 두 표현이 **자동으로 변환 가능한 상태**가 되는 것이다:

- 스펙에서 테스트 케이스가 자동 도출된다
- 테스트 실패가 스펙의 누락된 부분을 가리킨다
- 스펙 변경이 자동으로 테스트 업데이트로 이어진다

이 선순환이 완성되면 개발 파이프라인은 다음처럼 단단해진다:

**명세(의도) → 계획 → 작업 분할 → 구현 → 검증 → 명세 보완 → ...**

## AI 시대의 Red-Green-Refactor

전통적인 TDD 사이클을 AI 환경에서 재해석하면:

| 단계 | 전통 TDD | AI 시대 TDD |
|------|----------|-------------|
| **Red** | 개발자가 실패하는 테스트 작성 | 엔지니어가 명세에서 테스트 케이스 도출 (또는 AI에게 요청) |
| **Green** | 개발자가 테스트 통과 코드 작성 | AI 에이전트가 테스트를 통과하는 최소 코드 생성 |
| **Refactor** | 개발자가 설계 개선 | 에이전트가 아키텍처 일관성 유지하며 설계 개선 |

핵심 변화: **엔지니어의 역할이 코드 작성에서 기대 동작 정의로 이동한다.**

테스트 스위트는 AI가 작성한 코드의 건전성을 확인하는 체크포인트다. 코드를 일일이 읽는 대신, 테스트 통과 여부와 엣지 케이스 커버리지를 확인하는 것으로 리뷰 부담을 줄인다.

---

## AI 환각을 막는 가드레일

AI 에이전트의 고질적 문제는 **환각(Hallucination)** 이다. 틀린 API를 자신 있게 사용하거나, 존재하지 않는 함수를 호출하거나, 논리적으로 맞아 보이지만 동작하지 않는 코드를 만든다.

TDD가 이 문제의 가장 현실적인 방어막이다.

에이전트에게 구현을 맡기기 전에 기대하는 동작을 테스트 코드로 먼저 정의하면, 에이전트의 창의성을 요구사항이라는 범위 안에 가둘 수 있다. 테스트가 통과하지 않으면 구현이 완료된 것이 아니다.

이전에 우리가 ACP 브릿지 서버를 세 가지 방법으로 만들어본 실험에서도 확인했다. SDD v2(외부 스펙 참조 + 검증 단계 포함)만이 프로토콜 메서드명까지 정확하게 구현했다. 스펙이 정밀할수록 에이전트의 성공률이 급격히 상승한다.

---

## Spec-as-Source: 가장 급진적인 단계

SDD의 최종 형태는 **명세가 소스코드와 동일한 위상을 갖는 것**이다.

이 단계에서:
- 개발자는 코드를 직접 수정하지 않는다
- 명세를 수정하면 에이전트가 구현을 다시 생성한다
- 빌드 파이프라인이 명세-구현 일치를 강제 검증한다
- 불일치 → 빌드 실패

Architectural Drift가 구조적으로 불가능해진다.

---

## 개발자의 역할 변화

AI가 코드 작성을 대부분 처리하는 환경에서 인간 개발자의 핵심 역량은 다음으로 이동한다:

- **무엇이 "가짜 요구사항"인지 간파하는 능력** — 제품의 본질적 가치를 정의
- **엄격한 수용 기준(Acceptance Criteria) 설정** — AI의 논리적 허점을 짚어내고 "성공"을 정의
- **AI 결과물을 비판적으로 검토하는 능력** — 보안, 효율, 아키텍처 관점의 상위 수준 판단

한 가지 경고: AI 의존도가 높아지면 코드 레벨의 감각을 잃을 수 있다. "인지적 위축(Cognitive Atrophy)"이다. AI를 조수로 쓰되, AI가 만든 결과물을 판별할 수 있는 깊이는 유지해야 한다.

---

## 요점

Spec = TC는 단순한 테크닉이 아니다. 소프트웨어 개발에서 인간의 역할이 "구현자"에서 "의도 설계자"로 이동하는 패러다임 전환이다.

스펙을 정밀하게 쓰는 것이 곧 테스트를 정밀하게 쓰는 것이다. 그리고 그것이 AI 에이전트를 신뢰할 수 있게 만드는 유일한 방법이다.
