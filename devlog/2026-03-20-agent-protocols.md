# AI 에이전트 생태계의 표준화 흐름 — MCP부터 AG-UI까지

> 2026-03-20 | 리서치 노트

---

## 프로토콜 알파벳 수프

MCP, A2A, UCP, AP2, A2UI, AG-UI.

AI 에이전트 개발 생태계를 보면 약어들이 넘쳐난다. 이 목록을 처음 보면 경쟁하는 표준들의 벽처럼 느껴진다. 하지만 Google이 ADK(Agent Development Kit)를 통해 정리한 가이드를 보면, 이 프로토콜들은 **서로 경쟁하는 것이 아니라 레이어별로 분리된 문제를 각각 해결한다.**

레스토랑 공급망 에이전트를 예시로 각 프로토콜이 무엇을 하는지 보자.

---

## 6개 프로토콜, 6개 레이어

### MCP — 에이전트 ↔ 툴/데이터
에이전트가 외부 시스템에 연결할 때마다 커스텀 통합 코드를 작성하는 문제를 해결한다.

MCP 서버가 자신의 툴을 광고하면, 에이전트가 자동으로 발견한다. PostgreSQL, Notion, Mailgun — 각각 MCP 서버가 있으면 에이전트는 동일한 패턴으로 연결된다. 통합 코드 관리는 서비스를 만든 팀의 몫이 된다.

**핵심 패턴:** 표준화된 연결, 자동 발견

### A2A — 에이전트 ↔ 에이전트
에이전트가 다른 에이전트와 통신하는 문제. 서로 다른 팀이 만들고, 다른 프레임워크로 돌아가고, 다른 서버에 있다.

A2A 에이전트는 `/.well-known/agent-card.json`에 자신의 이름, 능력, 엔드포인트를 선언한다. 새 에이전트 추가는 URL 하나 추가하는 것으로 끝난다. 재배포도, 코드 수정도 없다.

**핵심 패턴:** `/.well-known/` URL 기반 자동 발견 + Agent Card

### UCP — 에이전트 ↔ 커머스
에이전트가 여러 공급업체에 주문을 넣어야 할 때, 각각 다른 API를 통합하는 문제. UCP는 쇼핑 라이프사이클을 typed request/response 스키마로 표준화한다. REST, MCP, A2A, EP — 어떤 전송 방식이든 동일한 패턴으로 동작한다.

### AP2 — 결제 권한 + 감사 추적
주문을 놓은 것은 UCP. 하지만 *누가 이 지출을 승인했는가?* 그 기록은 어디 있는가?

AP2는 `IntentMandate → PaymentMandate → PaymentReceipt` 흐름으로 결제 의도, 서명된 승인, 완료 영수증을 관리한다. 설정된 한도를 초과하면 서명 없이 진행되지 않는다.

### A2UI — 에이전트 → UI 렌더링
에이전트가 텍스트 대신 인터랙티브 UI를 만들어야 할 때. 18개 컴포넌트 primitive(Row, Column, Card, Button, TextField 등)로 선언형 JSON을 보내면, 클라이언트 렌더러(Lit, Flutter, Angular)가 네이티브 UI로 변환한다.

UI 구조와 데이터가 분리되어 있어, 데이터만 업데이트하면 컴포넌트를 다시 보낼 필요가 없다.

**핵심 패턴:** 제한된 primitive로 무한한 레이아웃 조합

### AG-UI — 에이전트 스트리밍 → 프론트엔드
에이전트의 응답은 REST API처럼 끝이 있지 않다. 텍스트를 점진적으로 스트리밍하고, 응답 중간에 툴을 호출하고, 사람의 입력을 기다린다. 이 복잡성을 처리하는 파싱 코드는 보일러플레이트다.

AG-UI는 이 보일러플레이트를 없앤다. `TEXT_MESSAGE_CONTENT`, `TOOL_CALL_START`, `TOOL_CALL_RESULT` 같은 typed 이벤트로 표준화된 SSE 스트림을 제공한다. 어떤 에이전트 프레임워크가 생성했는지와 무관하게 프론트엔드는 동일한 인터페이스를 쓴다.

## 스탠포드 OpenJarvis — 온디바이스로의 이동

같은 흐름에서 스탠포드가 발표한 OpenJarvis는 다른 방향을 가리킨다.

클라우드 중심 AI 에이전트를 **사용자 기기에서 직접 실행**하는 프레임워크다. 5계층 구조:

- **Intelligence**: 기기 성능에 맞는 LLM 자동 선택 (Qwen, Gemma, Phi 등 0.5~9B)
- **Engine**: Ollama, vLLM, llama.cpp 통합
- **Agents**: 오케스트레이터(계획/분할) + 오퍼레이티브(실행) 분리
- **Tools & Memory**: 사용자 데이터 로컬 보관
- **Learning**: 사용 이력 기반 자동 최적화

흥미로운 점은 성능 지표다. 대부분의 AI 프레임워크가 정확도만 측정하는 것과 달리, OpenJarvis는 **에너지 소비, FLOPs, 응답 지연, 쿼리당 비용**을 동시에 추적한다.

로컬 LLM과 AI 가속기가 88.7%의 경우에서 사용자 체감 속도를 높였다는 결과, AI 효율이 2023~2025년 사이 5.3배 향상됐다는 데이터가 이 방향을 뒷받침한다.

## SDD 관점에서 무엇이 중요한가

이 프로토콜들에서 반복되는 패턴이 있다:

**1. `/.well-known/` 기반 자동 발견**
A2A의 Agent Card, UCP의 프로필 — 둘 다 well-known URL로 capability를 선언한다. 에이전트가 스펙을 "발견"하고 자동으로 라우팅하는 구조다.

**2. Typed Schema**
각 프로토콜의 요청/응답은 강타입 스키마다. 자연어로 의도를 표현하는 것이 아니라, 기계가 검증할 수 있는 형태로 계약을 정의한다.

**3. 레이어 분리**
MCP는 툴 연결만, A2A는 에이전트 통신만, AP2는 결제 권한만. 각 레이어는 명확한 책임 범위를 가진다.

**4. 감사 추적**
AP2의 IntentMandate → PaymentMandate → PaymentReceipt 흐름은 "누가, 무엇을, 언제 결정했는가"를 추적한다.

이 4가지 패턴은 그대로 SDD 스펙 설계에 적용된다. 스펙도 자동으로 발견 가능해야 하고, typed 형태여야 하고, 레이어별로 분리되어야 하고, 결정 이력이 추적되어야 한다.

---

## 요점

AI 에이전트 생태계의 표준화 흐름이 보여주는 것은 하나다: **자연어와 커스텀 통합의 시대가 끝나고 있다.**

에이전트가 신뢰할 수 있게 동작하려면 각 레이어가 명확한 계약으로 연결되어야 한다. MCP, A2A, AP2는 이미 그 방향으로 수렴하고 있다.

스펙도 마찬가지다. "문서"가 아니라 "계약"이어야 한다.
